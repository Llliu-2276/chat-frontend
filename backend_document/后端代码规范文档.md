## 3. 分层架构规范

### 3.1 调用链

```
Controller → Service(接口) → ServiceImpl → Repository → Entity
                 ↕
             DTO / VO
```

### 3.2 各层职责与约束

| 层级 | 可以做什么 | 禁止做什么 |
|------|-----------|-----------|
| **Controller** | 接收请求、参数校验、调用Service、返回BaseResponse | ❌ 直接访问Repository<br>❌ 编写业务逻辑<br>❌ 返回Entity到前端 |
| **Service接口** | 定义业务方法签名 | ❌ 包含实现代码 |
| **ServiceImpl** | 实现业务逻辑、事务管理、调用Repository | ❌ 直接处理HTTP请求/响应<br>❌ 返回Entity到Controller |
| **Repository** | 数据库CRUD、自定义查询 | ❌ 编写业务逻辑<br>❌ 在循环中调用 |
| **Entity** | 映射数据库表、纯数据载体 | ❌ 返回给前端<br>❌ 包含业务逻辑 |

### 3.3 接口与实现分离

- ✅ **所有业务Service**必须定义接口+实现类（如 `UserService` + `UserServiceImpl`）
- ✅ 实现类放在 `service/impl/` 子包中
- **例外**：基础设施Service不需要接口，可以直接用 `@Service` 标注的类：
  - `TokenBlacklistService` — Token黑名单管理
  - `UserHeartbeatService` — 心跳状态管理
  - `RateLimitService` — 接口限流
  - `UserSessionManager` — WebSocket Session管理（`@Component`）

```java
}
```

### 4.2 方法级别规范

#### 4.2.1 方法结构

```java
@GetMapping("/path/{id}")
public BaseResponse<XxxVO> getXxx(@CurrentUser Long userId, @PathVariable Long id) {
    // 1. 日志记录
    log.info("获取xxx请求，userId: {}, id: {}", userId, id);
    
    // 2. 参数验证（复杂验证在 Service 层）
    
    // 3. 调用 Service
    XxxVO vo = xxxService.getXxx(userId, id);
    
    // 4. 返回统一响应（成功消息根据场景选择）
    return new BaseResponse<>(ErrorCode.SUCCESS.getCode(), "获取成功", vo);
}
```

#### 4.2.2 返回消息模板

| 场景 | code | message 示例 |
|------|------|-------------|
| 普通查询成功 | `SUCCESS.getCode()` | `"获取成功"` |
| 创建成功 | `CREATED.getCode()` | `"注册成功"` |
| 列表有数据 | `SUCCESS.getCode()` | `"获取xxx列表成功"` |
| 列表为空 | `SUCCESS.getCode()` | `"暂无数据"` |
| 操作成功 | `SUCCESS.getCode()` | `"操作成功"` |
| 删除成功 | `SUCCESS.getCode()` | `"删除成功"` |

#### 4.2.3 创建类操作

```java
@PostMapping("/create")
@ResponseStatus(HttpStatus.CREATED)         // ✅ 创建操作使用 201 状态码
public BaseResponse<XxxVO> createXxx(...) { ... }
```

#### 4.2.4 @CurrentUser 使用

```java
// ✅ 正确：用 @CurrentUser 自动注入当前用户ID
@GetMapping("/list")
public BaseResponse<Page<XxxVO>> getList(@CurrentUser Long userId) { ... }

// ❌ 错误：手动从 request 获取
// request.getAttribute("userId")  -- 不要在Controller中这样做
```

**关键点**：
- `@CurrentUser` 注入失败时会自动抛出 401 `BusinessException`，无需手动判空
- 公开接口（如 `GET /api/user/search/count`）**不使用** `@CurrentUser`，直接定义为无参方法

## 9. 异常处理规范

### 9.1 异常类层次

```
RuntimeException
  └── BaseException(code, message)          ← 抽象基类
        ├── BusinessException                ← 4xx 业务异常（WARN级别日志）
        │     ├── ForbiddenException         ← 403
        │     ├── ResourceNotFoundException  ← 404
        │     └── InvalidRequestParameterException ← 400
        └── SystemException                  ← 5xx 系统异常（ERROR级别日志）
              └── RemoteServiceException     ← 远程调用失败
```

### 9.2 抛出异常

```java
// ✅ 参数错误
throw new InvalidRequestParameterException(ErrorCode.BAD_REQUEST.getCode(), "消息内容不能为空");

// ✅ 资源不存在
throw new ResourceNotFoundException(ErrorCode.NOT_FOUND.getCode(), "用户不存在");

// ✅ 权限不足
throw new ForbiddenException(ErrorCode.FORBIDDEN.getCode(), "您不是该群的成员");

// ✅ 数据冲突
throw new BusinessException(ErrorCode.CONFLICT.getCode(), "你们已经是好友");

// ❌ 禁止直接抛出 RuntimeException 或 Exception
// throw new RuntimeException("xxx");
```

### 9.3 全局异常处理器

全局异常统一由 `ChatExceptionHandler`（`@ControllerAdvice`）处理，截获 **13种异常类型**：

| 异常类型 | HTTP状态码 | 日志级别 |
|---------|-----------|---------|
| `BusinessException` 及其子类 | 对应code | WARN |
| `SystemException` 及其子类 | 对应code | ERROR |
| `DataIntegrityViolationException` | 409 | WARN |
| `AuthenticationException` | 401 | WARN |
| `AccessDeniedException` | 403 | WARN |
| `BadCredentialsException` | 401 | WARN |
| `MethodArgumentNotValidException` | 400 | WARN |
| `IllegalArgumentException` | 400 | WARN |
| `NullPointerException` | 500 | ERROR |
| `NoResourceFoundException` | 404 | WARN |
| `Exception`（兜底） | 500 | ERROR |

**关键点**：兜底handler返回 `"系统繁忙，请稍后重试"`，不暴露内部错误细节。

## 13. 安全规范

### 13.1 密码安全

- ✅ 密码使用 **BCrypt** 加密存储
- ✅ 密码字段**永不**出现在日志中
- ✅ 密码字段**永不**出现在 API 响应中（`UserInfoVO` 不含 password）
- ✅ 修改密码后**所有旧Token自动失效**（Token版本控制机制）

### 13.2 JWT 认证

- JWT 有效期：**7天**（604800000ms）
- 密钥长度：**64字符**（生产环境应使用更安全的密钥）
- Token 传递：`Authorization: Bearer {token}`
- Token 黑名单：登出后加入 Redis 黑名单（TTL = Token剩余有效期）
- Token 版本控制：密码修改后 `token:version:{userId}` 递增

### 13.3 安全配置

```java
// 公开端点（无需认证）
.requestMatchers("/api/auth/login", "/api/auth/register").permitAll()
.requestMatchers("/ws/**").permitAll()  // WebSocket 由独立拦截器认证
.requestMatchers("/static/**", "/favicon.ico").permitAll()
.requestMatchers("/swagger-ui/**", "/swagger-ui.html", "/v3/api-docs/**").permitAll()

// 其他所有请求需要认证
.anyRequest().authenticated()
```

### 13.4 权限校验

- ✅ 群主权限：在 ServiceImpl 中手动校验（修改群信息、解散群、踢人、转让群主、审批入群）
- ✅ 好友关系：发消息前校验是否为好友
- ✅ 拉黑拦截：发消息/好友申请前校验是否被拉黑
- ✅ 自引用防护：不能给自己发消息、加自己为好友

### 13.5 SQL 注入防护

- ✅ 所有数据库操作通过 JPA Repository
- ✅ 没有手动拼接 SQL
- ✅ nativeQuery 使用 `:paramName` 参数化占位符

## 14. Redis 使用规范

### 14.1 Key 命名规范

实际使用的 Redis Key 模式：

| Key Pattern | 用途 | TTL |
|------------|------|-----|
| `user:login:{userId}` | 用户登录会话缓存 | 7天 |
| `user:heartbeat:{userId}` | 心跳时间戳 | 5分钟 |
| `user:online:{userId}` | 在线状态标记 | 5分钟 |
| `user:heartbeat:zset` | ZSET，所有在线用户的心跳时间 | 持久 |
| `token:blacklist:{token}` | Token黑名单 | Token剩余有效期 |
| `token:version:{userId}` | Token版本号（密码修改后递增） | 持久 |
| `rate:limit:{userId}:{apiPath}` | 接口限流计数 | 60秒 |

```java
// RedisTemplate 使用：
```
